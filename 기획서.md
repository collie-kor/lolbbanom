# 먼저봄 — 기획서

2026 전국 청소년 SW·AI 경진대회 본선 산출물 기획서
작성일: 2026.07.18

---

## 1. 주제

**지속가능한 도시와 공동체 — 휠체어·유모차 등 교통약자의 이동을 가로막는 장애**

## 2. 한 문장 정의

> 학교와 청소년 행사 공간의 사진 한 장에서 접근성 장애 요소를 찾아, 이용자 유형별로 다른 부담을 비교하고, 담당자가 당장 실행할 수 있는 최소 개선 요청문까지 만들어주는 웹 서비스.

## 3. 문제 정의

학교 축제를 준비할 때 부스 배치와 프로그램은 꼼꼼히 점검하지만, 휠체어나 유모차 이용자가 들어올 수 있는지는 점검 항목에 없다.

문제를 발견해도 "여기 턱 있네"라는 개인적 감상으로 끝난다. 행정실이나 행사 운영팀에 전달되지 않고, 전달되더라도 무엇을 어떻게 해달라는 것인지 구체화되지 않는다.

> **접근성 문제는 몰라서 안 고쳐지는 것이 아니다. 겪은 사람과 고칠 수 있는 사람이 서로 다른데, 그 사이를 잇는 말이 없어서 안 고쳐진다.**

## 4. 핵심 차별점

기존 접근성 서비스는 공간을 **접근 가능 / 불가능**이라는 하나의 값으로 판정한다.

먼저봄은 **같은 장애물이 이용자마다 전혀 다른 부담이 된다**는 것을 한 화면에서 보여준다.

| 5cm 턱 | 발생하는 부담 |
|---|---|
| 휠체어 이용자 | 독립적 진입 불가 |
| 유모차 이용자 | 들어 올려야 함, 동행자 필요 |
| 목발·보행 보조기 사용자 | 균형 상실 위험 |

> 접근성은 건물에 고정된 점수가 아니라, 공간과 사람 사이에서 발생하는 관계다.

이 관점 위에서, 확인된 장애물에 **직접 대응하면서 여러 이용자의 부담을 함께 줄이는, 가장 실행하기 쉬운 개선안**을 제안한다.

## 5. 기능 (3개 — 이외 구현 금지)

### 기능 1. 사진 기반 장애 요소 발견
- 공간 사진 1장 업로드
- AI가 장애 요소 후보 2~4개 추출
- 사용자가 `맞아요 / 아니에요 / 모르겠어요`로 검증
- **검증 단계는 절대 생략 불가**

### 기능 2. 이용자별 접근 비용 비교
- 휠체어 / 유모차 / 목발·보행 보조기 3열 비교
- 각 칸에 접근 비용 유형(이동·도움·정보·참여)과 설명 문장
- 하단에 실행 가능한 최소 개선안 1개 + 대안 1개
- **규칙표 기반 계산. AI 호출 없음**

### 기능 3. 위치 지정 + 수신자 맞춤 개선 요청문 생성
- **지도에서 해당 공간의 정확한 위치 선택** → 주소 자동 변환
- 수신자 선택 → 개선안 선택 → AI가 요청문 초안 생성
- 사용자가 수정·복사
- 하단에 **수신자별 전달·민원 접수 경로 안내**
- 자동 발송 버튼 없음

### 구현하지 않는 것
계정 · 로그인 · 제보 저장 · 지도 위 제보 누적 표시 · 경로 탐색 · 중복 판정 · 개선 전후 이력 · 통계 대시보드 · 자동 민원 제출 · PDF 생성 · 실제 기관 API 연동 · 앱스토어 배포

> 지도는 **위치 한 점을 찍어 주소 문자열을 얻는 용도로만** 쓴다. 마커가 쌓이는 제보 지도로 확장하는 순간 저장·중복 판정·운영 문제가 따라붙고, 18시간 안에 끝나지 않는다.

## 6. 서비스 흐름

```
사진 업로드
  ├─ 장애 요소 발견
  │    → 사용자 검증
  │    → 이용자별 접근 비용 비교
  │    → 최소 개선안 제시
  │    → 지도에서 위치 선택
  │    → 수신자별 요청문 초안
  │    → 전달 경로 안내
  │
  └─ 뚜렷한 장애 요소 없음
       → 관찰된 긍정 요소 표시
       → 추가 현장 확인 항목 안내
       → 점검 종료
```

**"문제 없음" 분기는 선택이 아니라 필수 기능이다.** 모든 사진에서 문제를 만들어내면 접근성 분석기가 아니라 비판 생성기가 된다. 잘 만들어진 공간에 "장애 요소가 관찰되지 않았습니다"라고 답할 수 있어야 이 서비스가 판정이 아닌 관찰을 한다는 것이 증명된다.

## 7. 규칙표 구조

### 장애물 6종
| ID | 이름 | 탐지 방식 |
|---|---|---|
| `curb` | 턱 또는 단차 | AI 추출 |
| `stairs_only` | 계단만 있는 출입구 | AI 추출 |
| `blocked_path` | 좁거나 막힌 통로 | AI 추출 |
| `obstruction` | 통로 위 임시 적치물 | AI 추출 |
| `heavy_door` | 무거운 여닫이문 | 사용자 직접 확인 |
| `no_signage` | 접근 가능 출입구 안내 부족 | 사용자 직접 확인 |

무료 티어 Flash 모델은 "문이 무거워 보인다" 같은 추론성 판단의 신뢰도가 낮다. 두 항목은 검증 화면에서 `사진으로는 확인이 어려운 항목입니다. 직접 확인해 주세요`로 물어본다. **모델의 한계를 신중한 설계로 전환한 지점.**

### 이용자 3종
`wheelchair` / `stroller` / `crutches`

### 접근 비용 4종
`mobility` 이동 · `assistance` 도움 · `information` 정보 · `participation` 참여

### 매트릭스 (18칸)
각 칸 구조:
```json
"curb_wheelchair": {
  "costs": ["mobility", "assistance", "participation"],
  "display": "바퀴가 단차를 넘기 어려워 혼자 진입하지 못하거나 다른 사람의 도움이 필요할 수 있습니다.",
  "phrase": "독립적 진입 불가 가능성",
  "applicable": true
}
```
- `display` — 화면 출력용 (`~할 수 있습니다` 형태로 통일)
- `phrase` — 요청문 프롬프트 주입용 명사구 (15자 이내)
- 해당 없는 칸도 키를 삭제하지 않고 `applicable: false`로 둔다

### 개선안 선정 로직 (수식 없음)
1. 사용자가 `맞아요`로 확인한 장애물만 사용
2. 해당 장애물에 직접 대응하는 개선안만 후보로 호출
3. 실행 난이도 낮은 순 정렬
4. 동률이면 실행 시점 빠른 순 정렬
5. 결과에는 **1순위 1개 + 대안 1개만** 표시

무거운 문에 경사판이 추천되지 않는 이유는 예외 규칙 때문이 아니라, 애초에 매핑에 없기 때문이다. 별도 예외 조항이 필요 없다.

**발표용 한 문장:** 확인된 장애물마다 실제로 대응 가능한 개선안만 연결하고, 그중 가장 쉽고 빠르게 실행할 수 있는 조치를 우선 제안합니다.

## 8. 전달·민원 접수 경로 안내

요청문 생성 화면 하단에 수신자별로 다르게 표시한다.

| 수신자 | 안내 내용 |
|---|---|
| 학교 행정실 / 담당 교사 | 직접 방문, 학교 홈페이지 건의 게시판, 학생회 안건 상정 |
| 행사 운영팀 / 학생회 | 준비 회의 안건 제출, 담당자 직접 전달 |
| 공공시설 (구청·주민센터 관할) | 국민신문고 또는 관할 지자체 민원 창구 |
| 민간 시설 (카페·상가 등) | 관리자에게 직접 전달. **민원 접수 대상이 아님을 명시** |

마지막 항목이 중요하다. 민간 상가의 턱을 국민신문고에 접수하면 처리되지 않는다. **"이건 민원 대상이 아닙니다"라고 알려주는 것도 안내다.**

- 수신자 ID → 안내 텍스트 매핑 JSON. API 호출 없음
- 외부 링크는 반드시 실제로 열리는지 확인 후 삽입
- `민원을 넣으면 해결됩니다` 같은 표현 금지. `전달할 수 있는 경로입니다` 수준으로 한정

## 9. 위치 지정 시스템

### 왜 필요한가
요청문에 **"어느 공간의 어느 출입구인지"**가 없으면 담당자가 현장을 찾아갈 수 없다. 민원 창구에 접수할 때도 위치는 필수 입력값이다. 사진만으로는 장소가 특정되지 않는다.

즉 지도는 기능 확장이 아니라 **요청문을 실제로 처리 가능하게 만드는 마지막 조각**이다.

### 범위 (엄격히 제한)
- 지도에서 핀을 찍거나 검색해서 **좌표 1개 선택**
- 좌표 → 도로명 주소 자동 변환 (reverse geocoding)
- 사용자가 세부 위치를 직접 보완 (`강당 정문`, `체육관 후문` 등)
- 이 세 값이 요청문 상단에 그대로 들어간다

**하지 않는 것:** 제보 마커 누적, 경로 탐색, 주변 시설 검색, 위치 저장

### 구현
- **Kakao Maps JavaScript SDK** 사용
  - 국내 도로명 주소 정확도가 가장 높고, 좌표↔주소 변환이 무료 제공
  - JavaScript 키는 **도메인 제한 방식**이라 노출되어도 등록된 도메인 외에서는 작동하지 않음
- 브라우저 Geolocation으로 현재 위치 기본값 제공, 사용자가 지도에서 수정
- 위치 미선택 시에도 요청문 생성은 가능하되, `위치 정보 없음` 경고 표시

### 사전 작업 (반드시 먼저)
1. Kakao Developers에서 앱 생성 → JavaScript 키 발급
2. **플랫폼 > Web 에 도메인 두 개 모두 등록**
   - `http://localhost:5173` (개발용)
   - `https://<프로젝트명>.vercel.app` (배포용)
3. 도메인 등록을 빠뜨리면 로컬에서는 되고 배포판에서만 지도가 안 뜬다. **가장 흔한 실패 지점.**

### 발표 시 주의
지도에 학교 위치를 정확히 찍으면 학교가 특정된다. 앞서 정한 "학교명 비공개" 원칙과 충돌한다.

→ 시연에서는 **지도 축척을 넓게 잡거나, 위치 화면을 짧게 넘긴다.** 기능이 작동한다는 것만 보이면 된다.

## 10. 배포 (Vercel)

### 구조
```
/
├── api/
│   ├── analyze.js      → Gemini 사진 분석 프록시
│   └── compose.js      → Gemini 요청문 생성 프록시
├── src/                → 프론트엔드
├── data/rules.json     → 규칙표
└── vercel.json
```

### API 키를 프론트엔드에 두지 않는다
Vercel Serverless Function을 프록시로 쓴다. 브라우저는 `/api/analyze`만 호출하고, Gemini 키는 서버 환경변수에만 존재한다.

- 환경변수명: `GEMINI_API_KEY` — **`VITE_` 접두사를 붙이면 안 된다.** 붙이는 순간 빌드 결과물에 그대로 박힌다
- Vercel 대시보드 > Settings > Environment Variables 에 등록
- Kakao JavaScript 키는 도메인 제한이 걸려 있으므로 프론트엔드에 있어도 무방

> 심사에서 "API 키 관리는 어떻게 했습니까"라는 질문이 나오면 이 구조를 그대로 설명하면 된다. 무료 티어에서 키를 노출하지 않고 배포한 것은 완성도 항목에서 유리하다.

### 배포 순서
1. GitHub 저장소 연결 → Vercel 자동 배포
2. 환경변수 등록 후 **재배포** (등록만 하면 반영 안 됨)
3. 배포된 도메인을 Kakao 플랫폼 설정에 추가
4. 배포판에서 사진 업로드 → 지도 → 요청문까지 **한 번 끝까지 돌려본다**

### 주의
- Serverless Function 기본 타임아웃 10초. 사진 분석이 길어지면 잘린다. 프롬프트를 짧게 유지하고 타임아웃 처리를 넣는다
- 제출 직전 배포 금지. **최소 2시간 전에 배포하고 실기기에서 확인**한다

## 11. UI 디자인 가이드

### 방향
참고한 두 사이트(행정 전문 법무법인)의 공통 특성을 따른다. 우리 서비스도 결국 **공적 문서를 만들어 담당자에게 전달하는 도구**이므로, 신뢰·행정 톤이 목적과 일치한다.

가져올 것:
- **그라데이션 없음.** 모든 배경은 단색 면
- 상단 짙은 단색 헤더 바
- 강한 대비의 굵은 한글 제목, 문장형 헤드라인
- 숫자·라벨 블록 (큰 값 위, 작은 설명 아래)
- 하단 고정 액션 바 (아이콘 + 짧은 라벨)
- 절제된 색 사용 — 강조색은 주요 행동 한 곳에만

버릴 것: 보라-파랑 그라데이션, 글래스모피즘, 둥근 blob 배경, 과도한 그림자, 이모지 아이콘

### 색상 토큰
```css
--ink:        #14263C;  /* 헤더 바, 제목 텍스트 */
--ink-soft:   #4A5A6E;  /* 보조 텍스트 */
--accent:     #A8202F;  /* 주요 행동 1개에만 */
--surface:    #FFFFFF;  /* 본문 배경 */
--panel:      #F2F4F6;  /* 구획 배경, 카드 */
--line:       #DCE1E6;  /* 1px 경계선 */
--flag:       #B45309;  /* 확인 필요 항목 */
```
강조색(`--accent`)은 화면당 한 곳. `공간 분석하기`, `요청문 만들기` 같은 최종 행동 버튼에만 쓴다.

### 타이포
- 본문·제목 모두 **Pretendard** (또는 시스템 `-apple-system, 'Malgun Gothic'`)
- 제목 700 / 소제목 600 / 본문 400
- 제목 24~28px, 본문 15~16px, 라벨 13px
- 자간 -0.02em (한글 굵은 제목 가독성)

### 형태
- `border-radius: 4px` (카드 6px). 완전한 원형·알약 형태는 아이콘 버튼에만
- 그림자 대신 **1px 경계선**으로 구획
- 카드 배경은 `--panel`, 본문은 `--surface`

### 시그니처 요소
**이용자별 3열 비교 화면.** 사진 한 장에서 세 개의 세로 열이 갈라져 나오고, 같은 장애물이 각 열에서 다른 문장으로 나타나는 순간이 이 서비스의 전부다. 여기에만 시각적 무게를 싣고 나머지 화면은 조용하게 유지한다.

### 진행 표시
상단에 `1 사진 — 2 확인 — 3 비교 — 4 요청문`. 현재 단계만 `--ink`, 나머지는 `--ink-soft`.

### 문구 원칙
- 판정하지 않는다: `접근 불가능합니다` ❌ → `사진에서 단차가 관찰됩니다. 현장 확인이 필요합니다` ⭕
- 버튼은 실행 결과를 그대로 말한다: `제출` ❌ → `요청문 만들기` ⭕
- 하단 고지: `사진 분석 결과는 공식 접근성 인증이나 법적 판정이 아닙니다.`

## 12. 기술 구성

- 반응형 웹, 단일 페이지 단계형 진행
- Gemini 무료 티어 Flash 계열 (분당 15회 / 일 1,500회) — **Vercel Serverless Function 경유**
- Kakao Maps JavaScript SDK (위치 선택, 좌표→주소 변환)
- AI 호출은 **두 곳뿐**: ① 사진 → 장애 요소 후보 ② 확인된 정보 → 요청문 초안
- 가운데 비교 매트릭스는 규칙표 JSON 계산. **API가 죽어도 작동**
- 배포: Vercel (GitHub 연동 자동 배포)
- 개발: Claude Code (메인) + Codex 리뷰
- Windows cmd 환경 — 한글 깨짐 시 `chcp 65001`

### 폴백 설계 (필수)
| 실패 지점 | 대체 경로 |
|---|---|
| 사진 분석 API 실패 | 체크박스로 장애물 직접 선택하는 경로로 전환 |
| 요청문 생성 API 실패 | 사전 작성 템플릿에 값만 주입 |
| 지도 SDK 로드 실패 | 주소·위치를 텍스트로 직접 입력 |
| 접근성과 무관한 사진 | 억지 지적 금지. `이 사진에서는 출입 동선이 확인되지 않습니다` |

### 개인정보
- 업로드 전 **필수 확인 단계**: 얼굴·개인정보·시설 식별 정보 제거 안내
- 무료 API 티어는 입력이 학습에 사용될 수 있음. 이 제약을 인지하고 설계에 반영했다는 점을 발표에서 명시

## 13. 구현 우선순위

시간이 부족하면 위에서부터 지킨다.

1. 사진 업로드 + 미리보기
2. AI 장애 요소 후보 추출 (Vercel Function 경유)
3. 사용자 확인·수정
4. **규칙 기반 이용자별 비교 매트릭스**
5. 최소 개선안 선정
6. AI 요청문 생성
7. 지도 위치 선택 + 주소 변환
8. 전달·민원 경로 안내
9. 디자인 마감, 애니메이션

요청문 생성보다 매트릭스가 먼저다. 요청문은 템플릿으로 대체 가능하지만, 이용자별 비교 화면이 없으면 이 서비스의 차별성 자체가 사라진다.

**배포는 별도 트랙이다.** 1~2번이 끝난 시점에 빈 껍데기라도 한 번 Vercel에 올려서 파이프라인을 뚫어놓는다. 마지막에 몰아서 배포하면 도메인 등록·환경변수 누락으로 시간을 잃는다.

| ② AI 협업 및 논리성 | AI는 사진 해석과 문장 생성만 담당. 이용자별 영향 판단은 검증 가능한 규칙표. 모델 한계와 데이터 정책까지 고려해 역할을 분리 |
| ③ 사회성 및 협업 | 문제를 발견하는 사람과 해결 권한이 있는 사람이 각자의 역할로 하나의 개선을 완성하는 구조 |

> 저희 서비스의 협업은 여러 사용자가 동시에 문서를 편집하는 기능이 아니라, 문제를 발견하는 사람과 해결할 권한이 있는 사람이 각자의 역할로 하나의 개선을 완성하는 과정입니다. 요청문은 최종 결과물이 아니라 그 협업을 시작시키는 첫 번째 대화입니다.

## 15. 발표 첫 30초

> 저희 학교 강당에는 정문 앞에 계단이 세 칸 있습니다. 작년 축제 때 유모차를 끌고 온 학부모가 입구에서 한참 서 있다가 결국 돌아갔습니다. 그걸 본 학생이 여러 명이었지만, 아무도 그 이야기를 행정실에 전하지 못했습니다. 어디에 어떻게 말해야 할지 몰랐기 때문입니다.
>
> 접근성 문제는 몰라서 안 고쳐지는 게 아닙니다. 겪은 사람과 고칠 수 있는 사람이 서로 다른데, 그 사이를 잇는 말이 없어서 안 고쳐집니다.
>
> 먼저봄은 공간 사진 한 장에서 시작합니다.

통계로 시작하는 발표는 수십 개가 나온다. 구체적인 장면 하나로 시작한다.

## 16. 데모 준비물

학교 사진 **3장** — 로직이 서로 다른 답을 내는지 검증하는 테스트 케이스이기도 하다.

1. 턱·계단이 있는 출입구 → 경사판 계열 개선안이 나와야 함
2. 통로에 물건이 쌓인 곳 → **적치물 즉시 이동**이 나와야 함 (여기서 경사판이 나오면 실패)
3. 문제가 없는 곳 → "장애 요소 없음" 분기 실연

촬영 시:
- 학교 간판·현판·교표가 프레임에 들어가지 않게
- 사람이 없는 시간대
- 발표에서는 "저희 학교"라고만 하고 학교명은 말하지 않는다

## 17. 절대 원칙 3개

1. **판정하지 않는다.** 관찰과 현장 확인 요청까지만.
2. **문제 없는 공간에는 문제가 없다고 답한다.** 억지 지적은 그 자리에서 신뢰를 무너뜨린다.
3. **자동으로 보내지 않는다.** 사용자가 사실을 확인하고 직접 복사해 전달한다.

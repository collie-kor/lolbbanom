# 먼저봄

> 공간 사진 한 장에서 접근성 장애 요소를 찾아, **같은 장애물이 휠체어·유모차·목발 이용자에게 만드는 서로 다른 부담**을 비교하고, 담당자에게 보낼 개선 요청문 초안까지 만들어 주는 웹 서비스.

## 🔗 배포 URL

### **https://lolbbanom.vercel.app/**

2026 전국 청소년 SW·AI 경진대회 본선 산출물.

## 해결하려는 문제

학교 축제나 행사를 준비할 때 부스와 프로그램은 꼼꼼히 점검하지만, 휠체어·유모차 이용자가 들어올 수 있는지는 점검 항목에 없습니다. 문제를 발견해도 "여기 턱 있네"라는 개인적 감상으로 끝나고, 행정실·운영팀에 전달되지 않거나 전달돼도 "무엇을 어떻게 해달라"가 구체화되지 않습니다.

**접근성 문제는 몰라서 안 고쳐지는 것이 아니라, 겪은 사람과 고칠 수 있는 사람 사이를 잇는 말이 없어서 안 고쳐집니다.** 먼저봄은 그 사이를 잇습니다.

---

## 핵심 차별점

기존 접근성 서비스는 공간을 **접근 가능 / 불가능** 이라는 하나의 값으로 판정합니다.

먼저봄은 **같은 장애물이 이용자마다 전혀 다른 부담이 된다**는 것을 한 화면에서 보여줍니다.

| 5cm 턱 | 발생하는 부담 |
|---|---|
| 휠체어 이용자 | 독립적 진입 불가 |
| 유모차 이용자 | 들어 올려야 함, 동행자 필요 |
| 목발·보행 보조기 사용자 | 균형 상실 위험 |

> 접근성은 건물에 고정된 점수가 아니라, 공간과 사람 사이에서 발생하는 관계입니다.

## 주요 기능

1. **사진 기반 장애 요소 발견**
   공간 사진 1장을 올리면 AI가 턱·계단·좁은 통로·적치물 등 장애 요소 후보를 찾고, 사용자가 `맞아요 / 아니에요 / 모르겠어요`로 검증합니다. AI는 정해진 4개 항목 외에 사진에서 관찰되는 다른 문제도 함께 제시하며, 목록에 없는 문제는 사용자가 **직접 입력**해 추가할 수 있습니다. 사진만으로 판단하기 어려운 항목(무거운 문·안내 부족)은 사용자가 직접 확인합니다. **검증 단계는 생략할 수 없습니다.**

2. **이용자별 접근 부담 비교**
   확인된 장애물이 휠체어·유모차·목발 이용자에게 각각 어떤 부담(이동·도움·정보·참여)이 되는지 3열로 비교하고, 가장 쉽게 실행할 수 있는 최소 개선안(1순위 + 대안)을 제시합니다. **이 계산은 규칙표 기반이라 AI 없이 작동합니다.**

3. **수신자 맞춤 개선 요청문 생성**
   위치(주소·세부 위치)와 수신자, 개선안을 고르면 AI가 요청문 초안을 만들고, 수신자별 전달·민원 접수 경로를 안내합니다. **자동으로 보내지 않으며**, 사용자가 직접 확인하고 복사해 전달합니다.

## 기술 스택

- **프론트엔드**: 순수 HTML·CSS·JavaScript (프레임워크 없음), 단계형 SPA
- **백엔드**: Node.js, Vercel Serverless Functions (AI 프록시)
- **AI**: Google Gemini API (무료 티어 Flash 계열)
- **로컬 개발 서버**: Express (배포에는 사용하지 않음 — Vercel 함수를 로컬에서 재사용하기 위한 용도)
- **배포**: Vercel (정적 `public/` + `api/` 서버리스 함수)
- **데이터**: 이용자별 부담·개선안·수신자 안내를 담은 규칙표 JSON (`public/data/rules.json`)

## 실행 방법

```bash
# 1. 의존성 설치
npm install

# 2. 환경변수 설정
cp .env.example .env        # Windows cmd: copy .env.example .env
# .env 파일을 열어 GEMINI_API_KEY 값을 채웁니다 (아래 '환경변수' 참고)
# 키가 없어도 폴백 모드로 끝까지 동작합니다.

# 3. 개발 서버 실행
npm run dev                 # 파일 변경 자동 반영 (npm start 도 동일하게 실행됨)
```

브라우저에서 **http://localhost:3000** 접속.

> Windows cmd에서 한글이 깨지면 `chcp 65001` 실행 후 다시 시도하세요.

### 환경변수

| 변수 | 용도 | 발급처 |
|---|---|---|
| `GEMINI_API_KEY` | 사진 분석 · 요청문 생성 | https://aistudio.google.com/apikey |
| `GEMINI_MODEL` | 사용할 모델 (선택, 기본 `gemini-flash-latest`) | — |
| `PORT` | 로컬 서버 포트 (선택, 기본 3000) | — |

키는 **서버 환경변수에만** 존재하고 브라우저에는 노출되지 않습니다. 브라우저는 `/api/*` 프록시만 호출합니다.

## 폴더 구조

```
api/                  Vercel 서버리스 함수 (AI 프록시)
  analyze.js          POST /api/analyze  사진 → 장애 요소 후보 (Gemini)
  compose.js          POST /api/compose  확인 정보 → 요청문 초안 (Gemini)
  config.js           GET  /api/config   AI 키 유무 전달
lib/
  core.js             API 공유 로직 (Vercel 함수 + 로컬 서버 공용)
public/               정적 프론트엔드 (배포 시 웹 루트)
  index.html          단계형 SPA (홈 → 사진 → 확인 → 비교 → 요청문)
  styles.css          디자인 토큰·레이아웃
  app.js              화면 전환·검증·규칙표 계산·요청문
  data/rules.json     장애물·이용자·접근 비용 매트릭스·개선안·수신자 안내
server.js             로컬 개발 서버 (api 핸들러 재사용, 배포에는 미사용)
vercel.json           배포 설정 (정적 + 서버리스 함수)
.env.example          환경변수 템플릿
기획서.md             기획 문서
```

AI 호출은 **두 곳뿐**입니다: `/api/analyze`, `/api/compose`.

## 설계 의도

### AI와 규칙표의 역할 분리

AI(Gemini)는 **① 사진 해석**과 **② 요청문 문장 생성** 두 곳에만 사용합니다. 정작 서비스의 핵심인 **이용자별 영향 판단은 규칙표(`rules.json`) 기반으로 계산**합니다.

이유는 두 가지입니다. 첫째, 무료 티어 모델은 "이 턱이 휠체어에게 얼마나 부담인가" 같은 판단의 신뢰도·일관성이 낮습니다. 같은 사진에 매번 다른 답을 내면 접근성 도구로 신뢰할 수 없습니다. 둘째, 이용자별 부담은 **검증 가능해야 하는 근거**입니다. 규칙표로 두면 "왜 이 개선안이 나왔는지"를 코드가 아니라 데이터로 설명할 수 있고, AI가 죽어도 이 부분은 그대로 작동합니다.

### API가 실패해도 끝까지 작동하는 폴백 구조

무료 API는 언제든 실패하거나 한도에 걸릴 수 있습니다. 그래서 모든 AI 지점에 대체 경로를 넣었습니다.

| 실패 지점 | 대체 경로 |
|---|---|
| 키 없음 / 사진 분석 실패 | 장애 요소를 직접 선택하는 체크박스 경로로 전환 |
| 요청문 생성 실패 | 사전 작성 템플릿에 값만 주입해 초안 생성 |
| 접근성과 무관한 사진 | 억지 지적 없이 "출입 동선이 확인되지 않습니다" 안내 후 수동 경로 |

가운데 **이용자별 비교와 개선안 선정은 애초에 규칙표 계산**이라 AI 상태와 무관하게 항상 동작합니다. 결과적으로 키가 하나도 없어도 사진 → 확인 → 비교 → 요청문까지 흐름이 끊기지 않습니다.

### 판정하지 않고 관찰만 하는 이유 · 자동 발송을 넣지 않은 이유

- **판정하지 않습니다.** 사진 한 장으로 "접근 불가능"이라고 단정하면, 잘 만들어진 공간을 오판하거나 현장과 다른 결론을 내는 순간 신뢰가 무너집니다. 그래서 "관찰됩니다 / 어려울 수 있습니다"라는 어조로만 말하고, 마지막 확인은 현장을 아는 사람에게 맡깁니다. 문제가 없는 공간에는 "장애 요소가 관찰되지 않았습니다"라고 답하는 분기도 필수 기능으로 넣었습니다.
- **자동으로 보내지 않습니다.** 요청문은 사람과 사람 사이의 협업을 시작시키는 첫 대화이지, 자동 제출용 민원이 아닙니다. 사용자가 내용을 직접 확인하고 복사해 전달하도록 해, 잘못된 내용이 담당자에게 그대로 나가는 일을 막습니다.
